我发现一个关键的规律：对于编程来说，它有很多种语言，包括 C++、Python、Go 以及其他语言。我弄这么多语言，从刚开始不熟悉，到能理解一些东西，也能自己构造。而且我结合了自己学外语的经历，还有对语言的研究——我觉得它们有共同之处。

## 第一步：先培养语感，而不是先背语法

学编程这种非常抽象的东西，它跟数学没有直接关系，更像是逻辑。

所以第一步是培养语感，需要看大量的 Python 代码、工程代码、汇编代码、C++ 代码。只有看得足够多，才会对现代工程设计的一些东西有感觉，比如异常处理、类的设计、防止出错的容错修复，以及方便编写者编写的语法糖。

这些语法糖你自己写代码时是没有感觉的，光靠背是不行的。尤其是异常处理，它非常重要，体现了防微杜渐的思想。但平常写作业或练习时，你很难有这个意识，因为不可能遇到那么多错误。这说白了属于冗余设计和保护设计，自己写代码时体会不到，需要足够地观察和阅读别人的代码。

这很像学语言：读英语不能只靠读几句，而是需要大量阅读，哪怕似懂非懂，也要培养语感。同时你需要"去魅"——无论是编程语言、英语还是中文，都要经历这个过程，你才会发现它其实没那么难，只是一种逻辑上的适应。

## 先学工程，再回头看语法

适应之后，可以开始学一些语法。但**不要先学完语法，再学数据结构，再学设计**。我建议反过来：先学工程，通过好的工具看到足够多好的代码，自己能做重构、重写等重要工作。有了这种感觉后，再回去看语法，就会对语法规则更有感觉，知道它怎么做了。

以我个人的经历来讲，如果一开始学 C++，其实根本没学到 C++。学校先教 C，我觉得特别简单，尤其是"指针"这个概念，因为没学到底层、内存调用和计算机系统结构，所以指针对我来说毫无意义。

后来打算法竞赛，发现那本质上是通用逻辑，跟 C 或 C++ 关系不大。很多高级语法更像应试技巧，没什么意思。

之后我学了 CS106L，接触了很多 C++20 和 C++23 的内容，才逐渐对现代 C++ 设计有了感觉。虽然不说是我自己写的，但我看的代码应该有几万行，所以才有这种体会。现在看 C++ 和 Python 的代码，我都觉得很舒服，不算特别困难。

## 工程代码量大，靠迭代而不是模仿

后来学 C++，以及用 C++ 和 Python 学 PyTorch、NumPy 等底层计算，还有架构设计、前后端项目中的防护和 HTTPS 等，其实都是相通的。因为工程代码量很大，不能指望像做数学题那样从零开始写，然后写十几个文件，再录个视频让人跟着逐行抄写——包括连注释都照抄，这种做法非常不切实际。

最好的办法是自己动手构建，从十几个小文件开始，由小到大逐步迭代，而不是直接照搬别人的代码。

这也是 MVP 的概念——先做一个最小的东西出来。结构越简单、架构越简单，它的稳定性越好。在稳定性越好的基础上，不停提升迭代，加一个新功能测试，然后不停砍。

这其实是个 trade-off：
- 一方面是通过增加冗余来提升功能性和安全性；
- 另一方面，增加冗余后整个结构变得复杂，如果原来的架构就很冗余，会导致看不清真实架构长什么样，进而影响后期迭代开发及整体稳定性。

所以需要在设计一开始就勇于砍掉很多看似有用的东西，砍到足够精简时，再往上加模块和结构就会好很多。

## 关于学语言：英语同理

学英语最重要的前提是**有动力**。学阿拉伯语是为了看《古兰经》，学德语是为了读原文版《浮士德》。学英语的动力，则应来自于阅读原版英语教材、新闻、人物访谈，以及用英文发帖等实际需求。

有了动力，还要坚持实践，从而建立原生语言学习的正向反馈，这样背单词时才不会抵触，也会更乐于将所学分享给他人。

在中国，很多人因应试教育而产生一种自发的羞耻感。应试教育的本质是追求高分，考低分的人因此感到羞耻——这不仅源于自身压力，更来自周围人的压力，以及老师的导向。进入大学后，这种状态并未减弱，导致许多人不敢开口说英语，因为开口就会暴露口语水平不足。缺乏正向反馈，口语自然难以提升，只能死记硬背单词，最终呈现出"没有学会"的状态。

我觉得英语最重要的是要运用，比如写帖子、写 X、写作文，或者跟人交流；也可以经常看英文论文，时常多用。这跟代码是一样的——学什么代码，最重要的是把它重构、重写、应用出来。多写分析、多写报告博客，也不要害怕，主要是去魅。

---

总结一句话：**学编程和学语言，底层逻辑是一样的——先大量输入建立语感，再通过真实使用建立直觉，语法和规则在这个过程中自然内化，而不是靠背。**
